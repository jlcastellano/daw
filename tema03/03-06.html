Servidores Web

1.5. Protocolo HTTP vs HTTPS

Cuando navegas por internet en 2025, probablemente has notado que algunas URLs comienzan con http:// mientras que otras comienzan con https://. Esta pequeña diferencia es extremadamente importante para tu seguridad y privacidad online.

¿Qué es HTTPS?

HTTPS (HTTP Secure) es la versión segura de HTTP. Añade una capa de cifrado a la comunicación entre tu navegador y el servidor web, protegiendo la información que viaja entre ambos.

Diferencias fundamentales entre HTTP y HTTPS

HTTP (no seguro):

  • La información viaja en texto claro, sin cifrar.
  • Cualquier persona o sistema que intercepte la comunicación puede leer todo el contenido: contraseñas, números de tarjeta de crédito, mensajes personales, etc.
  • No garantiza que estés conectado al servidor legítimo (vulnerable a ataques de suplantación).
  • En 2025, la mayoría de navegadores marcan los sitios HTTP como "No seguros".

HTTPS (seguro):

  • La información viaja cifrada, protegida mediante criptografía.
  • Incluso si alguien intercepta la comunicación, solo verá datos ilegibles.
  • Verifica la identidad del servidor mediante certificados digitales.
  • Protege la integridad de los datos (garantiza que no han sido modificados en tránsito).
  • En 2025, es el estándar para prácticamente todos los sitios web profesionales.

Certificados digitales y autoridades certificadoras

Para establecer una conexión HTTPS, el servidor web necesita un certificado digital. Este certificado funciona como una especie de "documento de identidad" del sitio web, que certifica que realmente es quien dice ser.

Los certificados son emitidos por entidades llamadas Autoridades Certificadoras (CA, por sus siglas en inglés). Estas son organizaciones de confianza que verifican la identidad de los solicitantes antes de emitir certificados. Los navegadores mantienen una lista de CA en las que confían.

Cuando visitas un sitio HTTPS:

  • Si el certificado está firmado por una CA reconocida, el navegador confía en el sitio y muestra un candado cerrado en la barra de direcciones.
  • Si el certificado es autofirmado o está firmado por una CA no reconocida, el navegador mostrará una advertencia de seguridad.
  • Si el certificado ha caducado o tiene otros problemas, el navegador también alertará al usuario.

Certificados autofirmados

Es posible que un servidor web genere sus propios certificados (autofirmados), sin intervención de una CA. Estos certificados proporcionan cifrado, pero los navegadores no confiarán en ellos automáticamente y mostrarán advertencias de seguridad.

Los certificados autofirmados son apropiados para:

  • Entornos de desarrollo y pruebas.
  • Redes internas de empresas, donde los empleados pueden instalar el certificado manualmente en sus dispositivos.
  • Situaciones donde todos los usuarios conocen y confían explícitamente en el sitio.

No son apropiados para sitios web públicos en internet, ya que los visitantes verán advertencias alarmantes que dañarán la credibilidad del sitio.

En 2025, obtener certificados válidos es mucho más accesible que en el pasado, gracias a iniciativas como Let's Encrypt, que proporciona certificados gratuitos y automatizados.

¿Rendimiento de HTTPS vs HTTP?

Cifrar y descifrar información requiere procesamiento computacional, por lo que históricamente HTTPS era más lento que HTTP. Sin embargo, en 2025 esta diferencia es prácticamente imperceptible para los usuarios gracias a:

  • Hardware más potente en servidores y dispositivos.
  • Mejoras en los algoritmos de cifrado.
  • Optimizaciones en los protocolos (como HTTP/2 y HTTP/3, que funcionan mejor con HTTPS).
  • Aceleración por hardware de las operaciones criptográficas.

De hecho, debido a las optimizaciones de HTTP/2 y HTTP/3 (que requieren HTTPS), muchos sitios HTTPS son ahora más rápidos que si usaran HTTP.

¿Cuándo usar HTTP vs HTTPS?

En 2025, la respuesta es simple: usa HTTPS siempre que sea posible. Los motivos incluyen:

  • Protección de la privacidad de los usuarios.
  • Mejor posicionamiento en buscadores (Google favorece sitios HTTPS).
  • Requisito para muchas funcionalidades web modernas (geolocalización, notificaciones push, service workers).
  • Confianza del usuario (los navegadores alertan sobre sitios no seguros).

HTTP solo es aceptable en entornos muy específicos, como redes completamente cerradas o dispositivos IoT muy limitados.

¿Pueden coexistir HTTP y HTTPS?

Sí, un mismo sitio web puede ser accesible tanto por HTTP como por HTTPS. Sin embargo, la práctica recomendada en 2025 es:

  1. Configurar el sitio para HTTPS.
  2. Redireccionar automáticamente todo el tráfico HTTP a HTTPS (usando una redirección 301).
  3. Implementar HSTS (HTTP Strict Transport Security) para forzar a los navegadores a usar siempre HTTPS.

¿Cómo funciona HTTPS?

Cuando escribes una dirección HTTPS en tu navegador, el proceso es el siguiente:

  1. Resolución DNS: el navegador traduce el nombre de dominio (www.ejemplo.com) a una dirección IP, igual que con HTTP.
  2. Conexión al puerto 443: el navegador se conecta al servidor en el puerto 443 (el puerto estándar para HTTPS, mientras que HTTP usa el puerto 80).
  3. Negociación TLS/SSL (handshake): antes de intercambiar cualquier dato de la página web, cliente y servidor establecen una conexión segura:
    1. El navegador solicita la identidad del servidor.
    2. El servidor envía su certificado digital, que incluye su clave pública.
    3. El navegador verifica el certificado (comprueba que sea válido, no haya caducado, y esté firmado por una CA de confianza).
    4. Si el certificado es válido, el navegador y el servidor acuerdan un método de cifrado y establecen claves de sesión únicas para esta conexión.
    5. Opcionalmente, el servidor puede solicitar un certificado del cliente (autenticación mutua), aunque esto es poco común en navegación web normal.
  4. Comunicación cifrada: una vez establecida la conexión segura, toda la comunicación posterior (peticiones y respuestas HTTP) viaja cifrada. Nadie que intercepte la comunicación podrá leer su contenido.

Puertos personalizados

Aunque 80 es el puerto estándar para HTTP y 443 para HTTPS, puedes configurar un servidor para usar puertos diferentes. Si lo haces, debes especificarlo en la URL:

  • http://www.ejemplo.com:8080 (HTTP en puerto 8080)
  • https://www.ejemplo.com:8443 (HTTPS en puerto 8443)

Sin embargo, usar puertos no estándar tiene inconvenientes:

  • Los usuarios deben conocer y recordar el puerto.
  • Algunos firewalls corporativos pueden bloquear puertos no estándar.
  • Los buscadores pueden tener dificultades para indexar el sitio.

Por estas razones, se recomienda usar los puertos estándar (80 y 443) para sitios web públicos.

Cuando no especificas el puerto en una URL, el navegador usa automáticamente el puerto estándar del protocolo:

  • http://www.ejemplo.com es equivalente a http://www.ejemplo.com:80
  • https://www.ejemplo.com es equivalente a https://www.ejemplo.com:443
