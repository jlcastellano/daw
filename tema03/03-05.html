Servidores Web

1.4. Uso del servicio HTTP

Ahora que comprendemos los fundamentos de HTTP, profundicemos en los métodos (también llamados "verbos HTTP") que se utilizan para interactuar con los servidores web.

Método GET

Se utiliza para obtener o recuperar un recurso del servidor. Es el método que se usa cuando escribes una URL en el navegador o haces clic en un enlace. Las características importantes de GET son:

  • No debería modificar datos en el servidor (es un método "seguro" e "idempotente").
  • Los datos que enviamos al servidor van incluidos en la propia URL, mediante lo que se conoce como cadena de consulta o query string.
  • Las peticiones GET pueden guardarse en el historial del navegador y en marcadores.

Ejemplos:

  • http://www.ejemplo.com/productos.php — solicita la lista de productos.
  • http://www.ejemplo.com/buscar.php?termino=zapatos&color=negro — envía parámetros de búsqueda. La parte después del símbolo "?" es la cadena de consulta, donde termino=zapatos y color=negro son parámetros que la aplicación web procesará.

Método POST

Se utiliza para enviar datos al servidor que normalmente modificarán algo en el sistema (como crear un nuevo registro, procesar un formulario, subir un archivo). Las características de POST son:

  • Los datos se envían en el cuerpo de la petición, no en la URL, por lo que no son visibles directamente para el usuario.
  • No hay límite práctico en la cantidad de datos que se pueden enviar.
  • Las peticiones POST no se guardan en el historial del navegador.
  • Es apropiado para operaciones que modifican datos en el servidor.

Por ejemplo, cuando llenas un formulario de registro y lo envías, normalmente se usa POST para que tus datos (nombre, email, contraseña) viajen de forma más segura y privada.

Método HEAD

Permite obtener únicamente las cabeceras de respuesta de un recurso, sin que el servidor envíe el contenido completo. Esto es útil para:

  • Verificar si un recurso existe sin descargarlo.
  • Obtener metadatos sobre un recurso (tamaño, fecha de última modificación, tipo de contenido).
  • Mantener cachés actualizadas eficientemente.

Por ejemplo, un navegador puede usar HEAD para verificar si una imagen ha cambiado desde la última vez que la descargó, sin necesidad de descargarla completamente.

Más allá de la navegación web tradicional

Hoy en día, HTTP se utiliza extensamente en lo que conocemos como servicios web o APIs web. Un servicio web es una interfaz que permite a diferentes aplicaciones comunicarse entre sí a través de internet.

Por ejemplo, si desarrollas una aplicación móvil que publica contenido en Instagram, tu aplicación utilizará la API de Instagram mediante peticiones HTTP. No necesitas un navegador; tu código se comunica directamente con los servidores de Instagram usando HTTP.

En el contexto de servicios web RESTful (que es el estilo arquitectónico más popular en 2025), los métodos HTTP adquieren significados específicos:

  • GET: se utiliza para recuperar o leer información. Por ejemplo, obtener los datos de un usuario específico o listar todos los productos de una categoría.
  • POST: se utiliza para crear nuevos recursos. Por ejemplo, publicar un nuevo mensaje en una red social o crear un nuevo producto en una tienda online.
  • PUT: se utiliza para actualizar o reemplazar completamente un recurso existente. Por ejemplo, actualizar todos los datos de perfil de un usuario.
  • PATCH: se utiliza para actualizar parcialmente un recurso existente. Por ejemplo, cambiar solo el email de un usuario sin tocar el resto de sus datos.
  • DELETE: se utiliza para eliminar un recurso. Por ejemplo, borrar una publicación o eliminar un producto del catálogo.

Códigos de estado de respuesta HTTP

Después de realizar cualquier petición HTTP, el servidor genera una respuesta que incluye un código de estado. Estos códigos nos informan sobre el resultado de la operación. Se organizan en cinco categorías:

Códigos 1XX — Respuestas informativas

Indican que la petición se recibió y el proceso continúa. Son poco comunes en la navegación web cotidiana.

  • 100 Continue: indica que todo va bien hasta ahora y que el cliente puede continuar con la petición o ignorar esta respuesta si la petición ya se completó.

Códigos 2XX — Respuestas de éxito

Indican que la petición fue recibida, comprendida y procesada correctamente.

  • 200 OK: la petición fue exitosa. Este es el código que verás en la mayoría de las navegaciones web normales. El servidor devuelve el contenido solicitado.
  • 201 Created: la petición fue exitosa y se creó un nuevo recurso como resultado. Común en APIs cuando se crea algo nuevo (por ejemplo, un nuevo usuario).
  • 204 No Content: la petición fue exitosa pero no hay contenido para enviar en la respuesta. Útil cuando se elimina algo o se actualiza sin necesidad de devolver datos.

Códigos 3XX — Redirecciones

Indican que se necesita realizar una acción adicional para completar la petición, normalmente porque el recurso se ha movido a otra ubicación.

  • 301 Moved Permanently: el recurso se ha movido permanentemente a una nueva ubicación. El servidor incluye la nueva dirección en una cabecera llamada Location. Los navegadores y buscadores actualizarán sus referencias a esta nueva ubicación.
  • 302 Found: el recurso se ha movido temporalmente a otra ubicación. Similar al 301, pero indica que el cambio es temporal.
  • 304 Not Modified: el recurso no ha cambiado desde la última vez que se solicitó. Esto permite al navegador usar su versión en caché, ahorrando ancho de banda y tiempo de carga.

Otros códigos de redirección incluyen 303 See Other, 307 Temporary Redirect y 308 Permanent Redirect, cada uno con matices específicos en cuanto a cómo debe manejarse la redirección.

Códigos 4XX — Errores del cliente

Indican que hubo un problema con la petición enviada por el cliente.

  • 400 Bad Request: la petición está mal formada o contiene errores que el servidor no puede procesar. Por ejemplo, datos JSON inválidos en una API.
  • 401 Unauthorized: se requiere autenticación para acceder al recurso, y las credenciales proporcionadas son inválidas o no se proporcionaron. A pesar del nombre, realmente se refiere a "no autenticado".
  • 403 Forbidden: el servidor entendió la petición, pero se niega a autorizarla. El usuario puede estar autenticado, pero no tiene permisos para acceder a ese recurso específico.
  • 404 Not Found: probablemente el código de error más famoso. El servidor no puede encontrar el recurso solicitado. Esto puede ocurrir porque el recurso fue eliminado, la URL contiene un error, o nunca existió.
  • 405 Method Not Allowed: el método HTTP usado no está permitido para ese recurso. Por ejemplo, intentar hacer POST en un endpoint que solo acepta GET.
  • 429 Too Many Requests: el cliente ha enviado demasiadas peticiones en un período de tiempo determinado. Es un mecanismo de protección contra abusos.

Códigos 5XX — Errores del servidor

Indican que el servidor encontró un error al procesar una petición válida.

  • 500 Internal Server Error: el servidor encontró una situación inesperada que le impidió completar la petición. Es un error genérico cuando algo salió mal en el servidor pero no se puede ser más específico.
  • 502 Bad Gateway: el servidor está actuando como puerta de enlace o proxy hacia otro servidor, y recibió una respuesta inválida de ese servidor.
  • 503 Service Unavailable: el servidor no está disponible temporalmente, generalmente debido a mantenimiento o sobrecarga. El servidor puede incluir una cabecera Retry-After indicando cuándo intentar de nuevo.
  • 504 Gateway Timeout: similar al 502, pero específicamente indica que el servidor upstream no respondió a tiempo.
