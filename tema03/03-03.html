Servidores Web

1.2. El servicio web e Internet

Internet, tal como la conocemos en 2025, funciona gracias a la arquitectura de protocolos TCP/IP. Esta arquitectura es fundamental porque permite la comunicación entre equipos que se encuentran en redes diferentes, proporcionando una infraestructura flexible y robusta para el intercambio de información.

Cuando accedemos a un sitio web mediante un navegador, se inicia una serie de comunicaciones con un sistema remoto que resultan en el intercambio de paquetes de datos, permitiendo finalmente que veamos la página web en nuestra pantalla. Analicemos este proceso paso a paso:

  1. Petición del recurso: el navegador realiza una petición de un recurso al servidor web. Este recurso puede ser una imagen, un documento HTML, un archivo de estilos CSS, un script JavaScript, u otros tipos de contenido.
  2. Procesamiento en el servidor: el servidor web recibe la petición y la procesa. Procesar la petición puede implicar dos escenarios diferentes: servir un recurso estático (simplemente enviar el contenido de un archivo que está almacenado en el disco del servidor), o ejecutar una aplicación en el servidor que genere contenido dinámico (por ejemplo, consultar una base de datos y construir una página HTML personalizada con los resultados).
  3. Renderizado: el navegador web recibe el contenido del servidor y lo procesa mediante un procedimiento llamado renderizado. Durante el renderizado, el navegador interpreta el código HTML, aplica los estilos CSS, ejecuta el JavaScript necesario, y convierte toda esa información en la página visual que finalmente vemos en la pantalla.
  4. Recursos adicionales: si para completar la visualización de la página el navegador necesita recursos adicionales (como imágenes, archivos CSS, fuentes tipográficas, scripts JavaScript), realizará nuevas peticiones al servidor siguiendo el mismo proceso anterior. Es común que una sola página web requiera decenas de peticiones individuales para cargar completamente.

El protocolo que hace posible todo este proceso es HTTP. HTTP define cómo el navegador debe formular sus peticiones y cómo el servidor debe estructurar sus respuestas. Comprender HTTP es fundamental para entender y configurar adecuadamente un servidor web.

Sin embargo, el éxito de la web no se debe únicamente al protocolo HTTP. Otros factores cruciales han contribuido a su desarrollo:

  • El sistema DNS (Sistema de Nombres de Dominio): traduce nombres de dominio legibles por humanos (como www.ejemplo.com) en direcciones IP numéricas (como 192.168.1.1) que los ordenadores utilizan para identificarse en la red. Gracias al DNS, no necesitamos memorizar largas series de números para acceder a nuestros sitios favoritos.
  • El uso de URLs (Localizadores Uniformes de Recursos): las URLs proporcionan una forma estandarizada y amigable de nombrar y acceder a recursos en la red. Por ejemplo: https://www.ejemplo.com/productos/zapatos.
  • La creatividad de desarrolladores web: que con HTML, CSS, JavaScript y tecnologías del lado del servidor, han logrado crear experiencias web cada vez más sofisticadas, interactivas e impresionantes.

Profundicemos ahora en el concepto de URL. La mejor forma de entender una URL es analizando un ejemplo concreto y descomponiendo sus partes:

https://es.wikipedia.org/wiki/Localizador_de_recursos_uniforme

Esta URL está compuesta por las siguientes partes:

  • Esquema (https:): indica el protocolo utilizado para acceder al recurso. Los esquemas más comunes son http: (protocolo HTTP estándar), https: (protocolo HTTP seguro con cifrado), y ftp: (protocolo de transferencia de archivos). También existen esquemas especiales como tel: (para números de teléfono), mailto: (para direcciones de correo electrónico), y muchos otros.
  • Nombre de dominio o FQDN (es.wikipedia.org): identifica el servidor específico al que estamos accediendo. En este caso, se trata del servidor de Wikipedia en español.
  • Ruta al recurso (/wiki/Localizador_de_recursos_uniforme): especifica la ubicación exacta del recurso dentro del servidor. Es similar a la ruta de un archivo en nuestro ordenador.

Cuando una URL requiere autenticación (como en algunos servidores FTP), podemos incluir credenciales de diferentes formas:

  • ftp://usuario@ftp.ejemplo.es/directorio/ — donde "usuario" es el nombre de usuario.
  • ftp://usuario:contraseña@ftp.ejemplo.es/directorio/ — donde se especifican tanto el usuario como la contraseña, separados por dos puntos. Nota: esta forma no es recomendable por motivos de seguridad, ya que las credenciales quedan visibles en la URL.

Finalmente, cuando un servicio web utiliza un puerto diferente al estándar, podemos especificarlo en la URL de la siguiente manera:

http://www.ejemplo.com:8080/recurso — donde 8080 es el número de puerto específico que se utilizará para la conexión. Esto es útil cuando se ejecutan múltiples servicios web en el mismo servidor o cuando se utilizan puertos no estándar por razones de seguridad o configuración.
