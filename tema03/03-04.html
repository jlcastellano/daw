Servidores Web

1.3. El protocolo HTTP

El protocolo HTTP (Protocolo de Transferencia de Hipertexto) es el lenguaje que permite la transferencia de información entre un cliente web (como tu navegador) y un servidor web en internet. Es la tecnología que da vida a la World Wide Web tal como la conocemos hoy.

HTTP ha evolucionado significativamente desde su creación. Comenzó con la versión 0.9 (muy básica) y ha pasado por varias iteraciones hasta llegar a las versiones modernas. A principios de 2025, encontramos principalmente tres versiones en uso:

  • HTTP/1.1: definida en el RFC 2616, fue durante muchos años la versión más extendida y sigue siendo muy utilizada. Es la versión que muchos desarrolladores aprenden primero.
  • HTTP/2: introducida en 2015, ofrece mejoras significativas en rendimiento y eficiencia. A fecha de 2025, su adopción es cada vez mayor y muchos sitios web importantes ya la utilizan.
  • HTTP/3: la versión más reciente, que comenzó a implementarse ampliamente a partir de 2022. Utiliza QUIC en lugar de TCP como protocolo de transporte, ofreciendo conexiones más rápidas y confiables, especialmente en redes móviles.

Aunque HTTP/2 y HTTP/3 son cada vez más prevalentes, es importante conocer las características fundamentales de HTTP/1.1, ya que los conceptos básicos se mantienen en todas las versiones:

  • Es un protocolo de nivel de aplicación: está diseñado específicamente para que aplicaciones en diferentes sistemas se comuniquen entre sí, sin preocuparse de los detalles técnicos de cómo viajan los datos por la red.
  • Es un protocolo sin estado: cada petición se trata de forma independiente. El servidor no "recuerda" automáticamente peticiones anteriores del mismo cliente. Cada interacción es como si fuera la primera vez. Esto simplifica el protocolo pero requiere mecanismos adicionales (como cookies) para mantener sesiones de usuario.
  • Utiliza comunicación de petición-respuesta: el cliente solicita un recurso, el servidor responde. Es un modelo simple pero muy efectivo.
  • Emplea formato de texto legible: tanto las peticiones como las respuestas (en HTTP/1.1) están escritas en texto que los humanos pueden leer y entender. Esto facilita enormemente la depuración y el aprendizaje. HTTP/2 y HTTP/3 utilizan formato binario para mayor eficiencia.
  • No incluye autenticación integrada: el protocolo básico no requiere que los usuarios se identifiquen para solicitar recursos. La autenticación debe ser implementada por las aplicaciones web cuando sea necesaria.

Como podemos observar, HTTP es intencionalmente simple en su concepción básica. Esto significa que funcionalidades como la autenticación de usuarios, el mantenimiento de sesiones, o la gestión de permisos deben ser implementadas por las aplicaciones web que construimos sobre HTTP, ya que el protocolo en sí mismo no las proporciona.

Para el seguimiento de usuarios y la gestión de sesiones, se utilizan cookies. Las cookies son pequeños fragmentos de información que el servidor web envía al navegador, y que este almacena temporalmente. Cada vez que el navegador realiza una nueva petición al mismo sitio web, envía automáticamente estas cookies de vuelta al servidor. Esto permite a la aplicación web "recordar" información sobre el usuario entre diferentes peticiones.

Por ejemplo, las cookies pueden almacenar:

  • El identificador de sesión del usuario (que permite al servidor recuperar su información de sesión)
  • Preferencias de visualización (como el idioma elegido o el tema oscuro/claro)
  • Productos visualizados recientemente en una tienda online
  • El contenido del carrito de la compra

Veamos ahora en detalle cómo funciona el proceso de comunicación HTTP. Cuando un cliente web (por ejemplo, tu navegador) solicita un recurso, se siguen estos pasos:

  1. El cliente web establece una conexión con el servidor web.
  2. El cliente web envía una petición al servidor. Existen diferentes métodos HTTP para realizar peticiones, siendo GET y POST los más comunes. Veremos más métodos en el siguiente apartado.
  3. El servidor procesa la petición y genera una respuesta, que incluye información sobre el resultado de la operación, el tipo de contenido que se envía, su tamaño, y el contenido en sí mismo.

Para comprender mejor este intercambio, analicemos un ejemplo concreto. Supongamos que escribimos en nuestro navegador la siguiente URL: http://www.ejemplo.es/inicio/

Nuestro navegador establecerá una conexión con el servidor www.ejemplo.es y enviará una petición HTTP similar a esta:

GET /inicio/ HTTP/1.1
Host: www.ejemplo.es
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0
Accept: text/html,application/xhtml+xml,application/xml
Accept-Language: es-ES,es;q=0.9
Accept-Encoding: gzip, deflate
Connection: keep-alive

Desglosemos esta petición:

  • La primera línea indica el método HTTP (GET), el recurso solicitado (/inicio/), y la versión del protocolo (HTTP/1.1).
  • A continuación, el navegador incluye varias cabeceras de petición que proporcionan información adicional al servidor:
    • Host indica el dominio al que se dirige la petición
    • User-Agent identifica el navegador y sistema operativo del cliente
    • Accept especifica qué tipos de contenido puede procesar el navegador
    • Accept-Language indica los idiomas preferidos del usuario
    • Accept-Encoding señala qué métodos de compresión soporta el navegador
    • Connection indica cómo debe manejarse la conexión

El servidor procesará esta petición y responderá con algo similar a esto:

HTTP/1.1 200 OK
Date: Wed, 08 Jan 2025 10:30:00 GMT
Server: Apache/2.4.52 (Unix)
Content-Type: text/html; charset=UTF-8
Content-Length: 3542
Connection: keep-alive
Set-Cookie: session_id=abc123xyz; Path=/; HttpOnly

<!DOCTYPE html>
<html>
<head>
    <title>Página de inicio</title>
</head>
<body>
    <h1>Bienvenido</h1>
    ...
</body>
</html>

La respuesta también tiene varias partes:

  • La primera línea indica la versión del protocolo (HTTP/1.1), un código de estado (200) que indica que todo fue bien, y un mensaje descriptivo (OK).
  • Seguidamente aparecen las cabeceras de respuesta:
    • Date indica cuándo se generó la respuesta
    • Server identifica el software del servidor web
    • Content-Type especifica el tipo de contenido que se envía (HTML en este caso) y su codificación de caracteres
    • Content-Length indica el tamaño del contenido en bytes
    • Connection indica cómo se maneja la conexión
    • Set-Cookie envía una cookie al navegador para que la almacene
  • Después de las cabeceras, hay una línea en blanco, seguida del contenido real que se enviará al navegador (en este caso, el código HTML de la página).

La diferencia fundamental entre HTTP/1.1 y las versiones más recientes (HTTP/2 y HTTP/3) radica en la eficiencia:

HTTP/2 introduce:

  • Comunicación binaria en lugar de texto
  • Multiplexación (permite enviar múltiples recursos simultáneamente en una sola conexión)
  • Compresión de cabeceras (reduce el overhead de cada petición)
  • Priorización de recursos (permite indicar qué recursos son más importantes)

HTTP/3 va aún más lejos al utilizar QUIC sobre UDP en lugar de TCP, lo que proporciona:

  • Establecimiento de conexión más rápido
  • Mejor rendimiento en redes con pérdida de paquetes
  • Migración de conexión (la conexión se mantiene aunque cambies de red WiFi a datos móviles)

Estas mejoras hacen que las páginas web se carguen más rápidamente y que la experiencia de usuario sea más fluida, especialmente en dispositivos móviles y conexiones inestables.
