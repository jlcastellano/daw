Estructura de Directorios y Protocolo LDAP

Estructura de los Directorios

La estructura de este tipo de directorios puede ser centralizada o distribuida dependiendo del diseño que se prefiera, por lo que se analizan ambas opciones:

Centralizado

Aquí todas las consultas se dirigen a un solo servidor, quien responde a todas ellas. La ventaja es que no es necesario sincronizarlo con otros servidores, pero el principal problema es que si falla, el sistema queda sin validación. Es esencial realizar al menos una copia de seguridad del servicio de directorio.

Distribuido

Esta alternativa permite que la información esté repartida entre varios servidores que pueden responder a las consultas de los clientes. Los datos pueden estar divididos o replicados:

  • Información dividida: Los servidores involucrados en el servicio de directorio no contienen toda la información, sino una parte de ella.
  • Información replicada: Toda la información del directorio forma parte de todos los servidores que componen el conjunto del servicio de directorio.

Configuración óptima: La opción más adecuada para configurar y proporcionar servicio de directorio es una combinación de las opciones anteriores, es decir, una parte de la información estará replicada y otra dividida. Esto sería lo ideal para compensar los inconvenientes de una u otra opción.

Protocolo LDAP

LDAP (Lightweight Directory Access Protocol) aparece como una alternativa a DAP. Se trata de una serie de estándares definidos por IETF, disponibles en varios RFC. Las características más destacadas que contribuyeron al éxito de LDAP son las siguientes:

  • LDAP utiliza TCP/IP en lugar de los protocolos teóricos del modelo OSI, que requiere menos recursos y es más accesible.
  • LDAP representa la información mediante cadenas de caracteres en lugar de complejas estructuras ASN.1.
  • El modelo funcional de LDAP es más sencillo y ha eliminado las opciones raras usadas en X.500, por lo que es más intuitivo de entender e implementar.
  • LDAP define un protocolo para el contenido de los mensajes entre un cliente y el servidor propiamente dicho. Este protocolo permite acceder a las operaciones definidas por el cliente, las respuestas del servidor y los datos transportados en el mensaje.

Modelos del Estándar LDAP

Profundizando, se va a analizar el estándar LDAP desde diferentes perspectivas que componen el servicio de directorio:

Modelo de Información

Este modelo nos permite dar forma a la estructura de la información almacenada en LDAP. El dato básico en el directorio es una entrada que corresponde con un objeto del mundo real. Una entrada se compone de un conjunto de atributos, cada uno con un tipo y sus valores.

El tipo, como en cualquier lenguaje de programación, define la información a almacenar y los valores son la información en sí. Todos los atributos tienen un identificador llamado OID y una sintaxis que permite definir qué valores puede tener.

Se asemeja a la forma de almacenar la información en Windows Server 2012, que también se basa en LDAP.

Conceptos Básicos
  • Entrada: Representa un objeto del mundo real (un usuario, un ordenador, un grupo, etc.)
  • Atributos: Cada entrada tiene atributos (nombre, apellido, correo, teléfono, etc.)
  • Tipo: Define qué clase de información puede almacenar el atributo
  • Valores: Los datos reales almacenados
  • OID: Identificador único para cada tipo de atributo
Ejemplo Práctico

Un usuario "Juan Pérez" sería una entrada con atributos como:

  • cn (Common Name): Juan Pérez
  • mail: juan.perez@empresa.com
  • telephoneNumber: 912345678
  • department: Ventas
dn: cn=Juan Pérez,dc=empresa,dc=com

El "dn" (Distinguished Name) es la dirección única de esta entrada en el directorio.

Modelo de Referencia

Al momento de nombrar los datos LDAP se define cómo se organizan y referencian estos. Primero se definen las estructuras de cómo se organizan las entradas y luego se indica cómo referenciar o acceder a las mismas.
